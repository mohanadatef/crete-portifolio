Technical Scope Document
Custom Real Estate Website
Laravel Backend + React or Angular Frontend | Arabic & English | Blog | Projects | Landing Pages | Leads
Purpose: This document defines the agreed technical scope for frontend and backend teams to start implementation with clear boundaries, simple requirements, and no unnecessary complexity.
Item	Decision
Project Type	Small custom company website with admin dashboard
Languages	Arabic and English  for website , dashboard English only
Backend	Laravel REST API
Frontend	Angular
Database	MySQL
Admin Panel	Custom admin panel for content, projects, landing pages, blog, and leads
Main Goal	Show company information, projects, landing pages, collect leads, and manage content

Module	Description	Priority
Authentication	Admin login/logout and protected dashboard routes	Must Have
Dashboard	Simple overview: total projects, landing pages, blog posts, and leads	Must Have

Projects	Add, edit, delete, publish/unpublish real estate projects	Must Have
Landing Pages	Create multiple landing pages for campaigns with lead forms	Must Have

Real Estate Website — Remaining Issues After First Build
Laravel + Angular 18  •  reviewed against the actual codebase

1. Verdict & Progress
Good progress on plumbing: Sanctum auth is correctly wired, a rate limit protects the public lead route, a sitemap route and a GTM/SEO meta service exist, and a mail notification is dispatched on new leads. But the heaviest and most business-critical items remain untouched, and the code introduced new defects.
Round-1 scorecard: 1 Done, 4 Partial, 16 Open. On top of these, Round-2 code review found 8 new issues — including 2 Critical (P0): a likely lead-capture failure and non-functional spam protection.
Bottom line: the app is not yet safe to launch. Fix the two P0 code bugs first (they break or expose the lead pipeline), then close SEO-01 (SSR) and the data-model items, since those get harder the more content is added.
2. Top Priorities — Fix in This Order
1.	BUG-01 — Stop sending recaptcha_token into Lead::create; verify the form actually saves a lead end-to-end.
2.	SEC-05 — Make reCAPTCHA real: check success/score, make the token required, add the secret to .env.
3.	SEC-06 — Add SoftDeletes to leads + a confirm step (your own scope rule).
4.	SEC-07 — Add throttle to /admin/login.
5.	SEO-01 — Decide and implement Angular SSR / prerender for public pages (biggest business risk).

3. New Issues Found in the Code (Round 2)
These were not in Round 1 — they are defects or gaps in the code that was written. Each includes a suggested fix.
BUG-01  Lead capture / Bug
Severity	P0
Problem	In LeadController@store, recaptcha_token is included in the validated array and then passed to Lead::create($validated). leads has no recaptcha_token column, so Eloquent will attempt to write a non-existent column and throw "Unknown column 'recaptcha_token'" whenever the frontend actually sends the token — i.e. every real submission fails.
Impact	The core lead-capture flow is very likely broken in production the moment a real reCAPTCHA token is sent. Verify immediately by submitting the form end-to-end.
Location	app/Http/Controllers/LeadController.php — store()
Suggested fix:
$data = collect($validated)->except('recaptcha_token')->all();
$lead = Lead::create($data);
SEC-05  Security / Anti-spam
Severity	P0
Problem	The reCAPTCHA call is security theater: the response success/score is never checked (the code comment literally says "we will just proceed"), the token is nullable so it can be omitted entirely, and RECAPTCHA_SECRET_KEY is not in .env.example (defaults to 'dummy_secret_key').
Impact	There is effectively NO bot protection. Lead spam will flow freely despite the appearance of a control.
Location	app/Http/Controllers/LeadController.php + .env
Suggested fix:
$res = Http::asForm()->post('https://www.google.com/recaptcha/api/siteverify', [
  'secret' => config('services.recaptcha.secret'),
  'response' => $request->recaptcha_token,
])->json();
abort_if(!($res['success'] ?? false) || ($res['score'] ?? 0) < 0.5, 422, 'Failed spam check');
BE-06  Backend / Media module
Severity	P1
Problem	MediaController is an empty stub (// only) even though the Phase 8 commit message claims "Media Uploads". There is no upload/list/delete endpoint and the media table is never written to.
Impact	The Media module promised in the scope (Must Have) is not actually implemented; images live only as ad-hoc paths inside the projects JSON column.
Location	app/Http/Controllers/MediaController.php
Suggested fix:
// Implement store(): validate image|mimes:jpg,jpeg,png,webp|max:4096,
// store on 'public' disk, persist row in media table, return public URL.
SEC-06  Data integrity / Leads
Severity	P1
Problem	Leads are hard-deleted: LeadController@destroy calls Lead::destroy($id) and the model has no SoftDeletes. This directly violates the scope's own rule: "Leads should not be deleted by default unless admin confirms deletion."
Impact	Accidental or malicious deletion permanently loses revenue data with no recovery and no audit trail.
Location	app/Models/Lead.php + leads migration
Suggested fix:
// Migration: $table->softDeletes();
// Model:    use Illuminate\Database\Eloquent\SoftDeletes; (add the trait)
// Require an explicit confirm flag before destroy().
SEC-07  Security / Auth
Severity	P1
Problem	The /admin/login route has no rate limiting, unlike the public lead route. Brute-force and credential-stuffing attempts against the single admin account are unthrottled.
Impact	Admin account is exposed to automated password guessing.
Location	routes/api.php
Suggested fix:
Route::post('/admin/login', [AuthController::class,'login'])
    ->middleware('throttle:5,1');
SEC-08  Security / Models
Severity	P2
Problem	Every model uses protected $guarded = [] (fully unguarded mass assignment). It is contained today only because controllers pass $validated, but any future $request->all() becomes an injection vector.
Impact	Latent mass-assignment risk across all entities.
Location	app/Models/*.php
Suggested fix:
// Replace with explicit allow-lists, e.g.:
protected $fillable = ['slug','title_ar','title_en','...'];
BE-05  Spec conformance
Severity	P2
Problem	Validation is written inline in controllers. The scope explicitly required "Laravel Form Requests for all create/update actions."
Impact	Deviation from agreed scope; duplicated rules between store/update; harder to maintain.
Location	app/Http/Controllers/*
Suggested fix:
php artisan make:request StoreProjectRequest
// move the validate() rules into the Form Request classes.
CFG-01  Configuration
Severity	P2
Problem	The sales notification email (sales@crete.com) is hardcoded in the controller, and the reCAPTCHA secret falls back to a 'dummy_secret_key' default. Mail-from is null in .env.example.
Impact	Cannot change recipient without a code change; misconfiguration ships silently to production.
Location	.env / config + LeadController
Suggested fix:
// Move to config/settings: Mail::to(config('mail.sales_inbox'))
// Add RECAPTCHA_SECRET_KEY and MAIL_FROM_ADDRESS to .env.example
